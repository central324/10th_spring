 - ## Page와 Slice
    
    ### 1. Page (페이지)
    
    > **"전체 현황 파악이 중요한 일반 게시판형"**
    > 
    - **정의**: 전체 데이터의 양을 기반으로 현재 페이지의 데이터를 가져오는 인터페이스
    - **작동 원리**: 데이터 조회 쿼리와 별도로, 전체 레코드 개수를 가져오는 **`count` 쿼리**를 추가 실행
    - **주요 제공 메서드**:
        - `getTotalElements()`: 전체 데이터 개수 반환
        - `getTotalPages()`: 전체 페이지 수 반환
        - `getContent()`: 현재 페이지의 데이터 목록
    - **장점**: 사용자가 전체 규모를 파악할 수 있고, 특정 페이지 번호(예: 7페이지)로 직접 이동이 가능함
    - **단점**: 데이터가 수백만 건 이상일 경우 `count` 쿼리 실행으로 인한 성능 저하가 발생할 수 있음
    - **적합한 UI**: 일반적인 웹 사이트 게시판, 관리자 페이지 등
    
    ### 2. Slice (슬라이스)
    
    > **"성능 최적화가 중요한 무한 스크롤형"**
    > 
    - **정의**: 전체 개수 조회 없이 다음 페이지의 존재 여부만 확인하는 인터페이스
    - **작동 원리**: `count` 쿼리를 실행하지 않음. 대신 요청한 사이즈(`size`)보다 **1개 더 많은 데이터(`size + 1`)를 조회**하여 다음 데이터가 있는지 확인
    - **주요 제공 메서드**:
        - `hasNext()`: 다음 페이지 존재 여부 (boolean)
        - `hasPrevious()`: 이전 페이지 존재 여부 (boolean)
        - `getContent()`: 현재 슬라이스의 데이터 목록
    - **장점**: 불필요한 `count` 쿼리가 나가지 않아 데이터 양이 많을수록 **성능 최적화**에 유리함
    - **단점**: 전체 데이터 개수나 마지막 페이지 번호를 알 수 없음
    - **적합한 UI**: 모바일 앱의 **무한 스크롤(Infinite Scroll)**, SNS의 '더 보기' 버튼 등
    
    | **비교 항목** | **Page** | **Slice** |
    | --- | --- | --- |
    | **Count 쿼리** | **수행함 (성능 오버헤드 있음)** | **수행 안 함 (성능 최적화)** |
    | **상속 관계** | `Slice`를 상속받아 확장함 | 페이징의 기본 인터페이스 |
    | **데이터 조회량** | 요청한 `size` 만큼 조회 | 요청한 `size + 1` 만큼 조회 |
    | **주요 정보** | 전체 개수, 전체 페이지 번호 | 다음 페이지 존재 여부 (`hasNext`) |
    | **추천 사용처** | 번호 기반 페이징 UI | 무한 스크롤, 더 보기 UI |
   
- ## Java stream API
    
    ### 1. 정의 및 핵심 개념
    
    > **"데이터의 흐름(Stream)을 선언적으로 처리하는 도구"**
    > 
    - **정의**: 자바 8부터 도입된 기능으로, 컬렉션(Collection)이나 배열의 데이터를 마치 공장의 **파이프라인**처럼 하나씩 흘러보내며 가공하는 인터페이스입니다.
    - **핵심 철학**: '어떻게(How) 코드를 짤 것인가'가 아닌, '무엇을(What) 할 것인가'에 집중하는 **선언적 프로그래밍**을 지향합니다.
    
    ### 2. 스트림의 3단계 구조 (Pipeline)
    
    스트림은 한 번 생성되면 다음과 같은 파이프라인을 거쳐 소모됩니다.
    
    1. **생성 (Stream Creation)**: 컬렉션(List, Set)이나 배열을 스트림 형태로 변환합니다.
        - 예: `list.stream()`
    2. **중간 연산 (Intermediate Operations)**: 데이터를 가공하거나 필터링합니다. 결과가 다시 스트림이므로 연결(Chaining)이 가능합니다.
        - 핵심 메서드: `filter()`, `map()`, `sorted()`, `distinct()`, `limit()`
    3. **최종 연산 (Terminal Operations)**: 가공된 데이터를 소모하여 최종 결과(결과값, 컬렉션, 출력 등)를 만듭니다. 이 연산이 호출되면 스트림은 닫힙니다.
        - 핵심 메서드: `collect()`, `forEach()`, `count()`, `reduce()`, `anyMatch()`
    
    ### 3. 주요 특징
    
    - **원본 데이터 불변성 (Read-only)**: 스트림은 데이터를 읽기만 할 뿐, 원본 컬렉션의 데이터를 변경하지 않습니다.
    - **일회용 (Lazy Evaluation)**: 한 번 사용한 스트림은 다시 재사용할 수 없습니다(재사용 시 에러 발생).
    - **지연 연산**: 최종 연산이 호출되기 전까지 중간 연산은 실제로 실행되지 않습니다. 이를 통해 최적화된 연산을 수행합니다.
    - **병렬 처리 용이성**: `parallelStream()`을 사용하면 내부적으로 ForkJoinFramework를 활용해 대용량 데이터를 손쉽게 병렬로 처리할 수 있습니다.

- ## 객체 그래프 탐색 (Object Graph Traversal)

### 1. 정의 및 핵심 개념

> **"참조를 따라 연관된 객체들을 찾아가는 과정"**
> 
- **정의**: 객체 지향 프로그래밍에서 어떤 한 객체를 시작점으로 하여, **연관된 객체들의 연결망(Graph)을 참조를 통해 따라가며 데이터에 접근하는 것**을 의미합니다.
- **핵심 동작**: `member.getTeam()` 또는 `order.getMember().getAddress()`와 같이 마침표(`.`) 연산자를 이용해 꼬리에 꼬리를 무는 방식으로 객체를 탐색합니다.

### 2. JPA에서의 객체 그래프 탐색

데이터베이스의 테이블은 외래 키(FK)로 관계를 맺지만, 자바 객체는 **참조**로 관계를 맺습니다. JPA는 이 간극을 메워 객체 그래프 탐색을 가능하게 합니다.

- **자유로운 탐색의 제약**: 이론적으로는 객체 그래프를 어디든지 탐색할 수 있어야 하지만, 실제로는 데이터베이스에서 객체를 조회할 때 연관된 모든 객체를 한꺼번에 다 가져오는 것은 성능상 불가능합니다.
- **해결책 (지연 로딩)**: JPA는 **지연 로딩(Lazy Loading)** 기법을 통해 실제 객체가 사용되는 시점에 SQL을 실행하여 객체 그래프 탐색을 지원합니다.

### 3. 주요 특징 및 주의사항

1. **객체 그래프의 경계**: 데이터베이스에 저장된 모든 객체를 메모리에 올릴 수 없으므로, 처음 SQL을 통해 어디까지 조회하느냐에 따라 탐색할 수 있는 '경계'가 정해집니다.
2. **Entity Manager의 관리**: JPA에서 객체 그래프 탐색이 정상적으로 동작하려면 해당 객체가 영속성 컨텍스트(Persistence Context)에 의해 관리되는 상태여야 합니다. (준영속 상태에서는 탐색 시 에러 발생 가능)
3. **성능 최적화 (Fetch Join)**: 꼬리에 꼬리를 무는 탐색 중 발생하는 **N+1 문제**를 해결하기 위해, 한 번의 쿼리로 연관 객체까지 가져오는 `Fetch Join`이 필수적으로 사용됩니다.

### 객체 vs 데이터베이스

| **구분** | **객체 그래프 탐색** | **SQL 조인 (Join)** |
| --- | --- | --- |
| **접근 방식** | 참조 (`.`)를 이용한 선형/그물망 탐색 | 외래 키(FK)를 이용한 테이블 간 결합 |
| **방향성** | 참조가 있는 방향으로만 가능 (단방향/양방향) | 외래 키만 있으면 양방향 조회 가능 |
| **시점** | 실행 시점에 필요한 객체를 따라감 | 쿼리 작성 시점에 조회 대상이 결정됨 |

- ## @Valid vs @Validated
     
  ### 1. 정의 및 핵심 개념
    
    > **"입력 데이터의 유효성을 검증하는 두 가지 방식"**
    > 
    - **@Valid**: 자바 표준 스펙(JSR-303, Bean Validation)에서 제공하는 어노테이션입니다.
    - **@Validated**: Spring 프레임워크에서 제공하는 어노테이션으로, **@Valid의 기능을 확장**하여 추가적인 기능을 제공합니다.
    
    ### 2. 주요 차이점
    
    ### **(1) 검증 그룹 (Validation Grouping)**
    
    - **@Valid**: 그룹화 기능을 지원하지 않습니다. 모든 필드를 한꺼번에 검증합니다.
    - **@Validated**: **그룹화 기능**을 지원합니다. 예를 들어, '등록'할 때와 '수정'할 때 서로 다른 검증 규칙을 적용해야 할 경우 `@Validated(CreateGroup.class)`와 같이 지정하여 사용 가능합니다.
    
    ### **(2) 검증 위치 (Scope)**
    
    - **@Valid**: 주로 컨트롤러(Controller) 메서드의 **Request Body나 객체 파라미터** 검증에 사용됩니다.
    - **@Validated**: 컨트롤러뿐만 아니라 **서비스(Service) 레이어** 등 다른 계층에서도 검증이 가능합니다. 클래스 레벨에 선언하여 해당 클래스의 메서드 파라미터들을 검증할 수 있습니다.
    
    ### **(3) 예외 발생 종류**
    
    - **@Valid**: 검증 실패 시 `MethodArgumentNotValidException`이 발생합니다.
    - **@Validated**: 메서드 수준에서 검증 실패 시 `ConstraintViolationException`이 발생합니다.
    
    ### 3. 비교하기
    
    | **비교 항목** | **@Valid** | **@Validated** |
    | --- | --- | --- |
    | **제공처** | Java 표준 (JSR-303/JSR-380) | Spring Framework |
    | **주요 용도** | 컨트롤러 단 단순 객체 검증 | 계층 제약 없는 검증, 그룹 검증 |
    | **그룹핑 지원** | ❌ 지원 안 함 | ✅ 지원함 (Groups 활용) |
    | **검증 위치** | 메서드 파라미터 (객체 위주) | 클래스 레벨, 메서드 파라미터 |
    | **Spring 의존성** | 낮음 (자바 표준) | 높음 (Spring 전용) |
    
    ### 4. 사용 가이드라인
    
    1. **일반적인 REST API 컨트롤러**: 단순히 RequestBody를 검증할 때는 자바 표준인 **@Valid**를 사용하는 것이 관례입니다.
    2. **그룹별 검증 필요 시**: 등록 시에는 필수지만 수정 시에는 필수가 아닌 필드가 있다면 **@Validated**를 사용해 그룹을 나눕니다.
    3. **서비스 레이어 검증**: 컨트롤러가 아닌 곳에서 파라미터(`@NotBlank` 등)를 검증해야 한다면 클래스 상단에 **@Validated**를 붙여야 동작합니다.
