## 전체 구현
이번 미션에서는 먼저 엔티티를 설계하고 엔티티 간 연관관계를 매핑한 뒤,
이를 바탕으로 미션 조회 / 상세 조회 / 도전 시작 / 리뷰 작성 / 특정 지역에서 도전 가능한 미션 조회 / 내가 진행 중인 미션 조회 기능을 구현했다.

또, 5주차에 작성했던 컨트롤러와 서비스 계층을 연결해서,
실제 요청이 들어왔을 때 컨트롤러가 서비스를 호출하고, 서비스가 비즈니스 로직을 수행한 뒤 응답을 반환하도록 구성했다.

특히 이번 미션에서는 조회 기능에서 요구된 페이징을 단순 메모리 처리 방식이 아니라,
JPA의 @Query를 활용한 조회 방식으로 구현하여 데이터베이스 레벨에서 필요한 데이터만 가져오도록 구성했다.

## 엔티티 설계
도메인 구조를 엔티티로 분리하고 관계를 명확히 설정하는 작업을 먼저 진행하였다.

예를 들어, 서비스 요구사항을 보면 다음과 같은 관계가 자연스럽게 존재한다.

회원(Member) 은 여러 미션에 도전할 수 있다.
회원(Member) 은 여러 리뷰를 작성할 수 있다.
가게(Store) 는 여러 미션을 가질 수 있다.
가게(Store) 는 여러 리뷰를 가질 수 있다.
미션(Mission) 에 대해 회원이 도전한 기록은 별도의 엔티티로 관리할 수 있다.
이런 구조를 단순히 하나의 엔티티에 모두 넣는 방식이 아니라,
각 역할을 분리해서 엔티티를 작성하였다.

### 역할 분리
엔티티 하나가 너무 많은 책임을 가지면 유지보수가 어려워지기 때문에,
각 도메인의 역할에 맞춰 Member, Store, Mission, Review, MemberMission 같은 식으로 분리했다.

### 연관관계 명확화
예를 들어 리뷰는 “누가 작성했는지”, “어느 가게에 대한 리뷰인지”가 중요하므로
리뷰 엔티티에서 회원과 가게를 참조하도록 매핑했다.
이렇게 하면 리뷰 조회 시 작성자나 가게 정보를 자연스럽게 연결해서 사용할 수 있다.

### 확장성 고려
미션 도전 여부를 단순히 Member나 Mission에 boolean 값으로 두는 방식도 가능하지만,
그렇게 하면 여러 회원이 여러 미션에 도전하는 관계를 관리하기 어려울 수 있다는 생각이 들었다.
그래서 중간 엔티티를 두어 도전 상태, 진행 여부, 시작일 등 추가 정보를 확장할 수 있도록 설계했다.


## 진행 중인 미션 조회
이 기능은 특정 회원이 현재 진행 중인 미션들을 조회하는 기능이다.

이때 단순히 전체 미션을 가져온 뒤 애플리케이션 메모리에서 필터링하는 방식 대신,
DB 조회 시점에서 진행 상태인 데이터만 가져오도록 설계했다.

이렇게 한 이유는 다음과 같다.

- 불필요한 데이터 조회를 줄일 수 있다.
- 데이터가 많아질수록 성능상 유리하다.
- 페이징 처리와 결합하기 쉽다.

또한 이 기능은 사용자가 한 번에 많은 미션을 가질 수도 있기 때문에
페이징이 필요하다고 판단했고, 이를 @Query 기반으로 조회하도록 구성했다.


## 해당 지역에서 도전 가능한 미션 조회
이 기능은 특정 지역에 속한 가게들의 미션 중 사용자가 도전 가능한 미션을 조회하는 기능이다.

여기서 핵심은 단순히 미션 전체를 보여주는 것이 아니라,
지역 조건을 기준으로 필터링해서 필요한 데이터만 보여주는 점이다.

이 기능을 구현하면서 신경 쓴 부분은 다음과 같다.

- 가게와 지역의 관계를 기준으로 미션을 조회할 수 있도록 엔티티 관계를 활용
- 사용자가 실제로 필요한 범위의 데이터만 볼 수 있도록 조회 조건을 제한
- 결과가 많아질 수 있기 때문에 페이징 적용
즉, 조회 조건이 분명한 API이므로
Repository에서 @Query를 활용해 필요한 조건을 직접 명시하고,
서비스에서는 이 쿼리 결과를 응답 형태로 가공하는 구조로 구현했다.


## 미션 도전 시작 기능
이 기능은 사용자가 특정 미션에 도전하기 시작했음을 기록하는 기능이다.

이 부분은 단순히 “도전했다”는 값만 바꾸는 것이 아니라,
회원과 미션 사이의 관계를 생성하는 것에 가깝다고 판단했다.
그래서 별도의 도전 관련 엔티티 MemberMission를 활용하는 방식이 적절했다.

이 구조를 사용한 이유는 다음과 같다.

- 한 명의 회원은 여러 미션에 도전할 수 있다.
- 하나의 미션은 여러 회원이 도전할 수 있다.
- 도전 상태, 도전 시작일, 성공 여부 같은 부가 정보를 저장할 수 있다.
즉, 단순 다대다 관계로 끝내지 않고
실제 서비스 로직에 필요한 정보를 담을 수 있는 구조로 설계했다는 점이 핵심이다.

## 미션 상세 조회 기능
미션 상세 조회는 특정 미션 하나를 선택했을 때
그 미션의 세부 정보들을 확인하는 기능이다.

이 기능은 비교적 단순한 조회 기능이지만,
단건 조회이기 때문에 잘못된 ID 요청이 들어왔을 때의 예외 처리가 중요했다.

따라서 구현 시에는 다음을 고려했다.

- 해당 미션이 존재하는지 확인
- 존재하지 않는 경우 적절한 예외 처리
- 존재하는 경우 필요한 정보만 응답 DTO로 반환
여기서도 엔티티를 그대로 외부에 노출하기보다
응답 DTO로 변환해서 반환하는 구조를 사용했다면,
이는 API 응답 구조를 안정적으로 관리하기 위함이다.

## @Query 사용
이번 미션 요구사항에서 페이징 부분은 @Query를 통해 구현하도록 되어 있었기 때문에,
단순 메서드 네이밍 쿼리만 사용하는 것이 아니라 직접 조회 조건을 명시한 JPQL 쿼리 방식을 사용했다.

이렇게 한 이유는 다음과 같다.

- 복잡한 조회 조건 표현이 쉬움
진행 중 미션 조회나 특정 지역 미션 조회처럼 연관 엔티티를 따라가며 조건을 주어야 하는 경우,
메서드 이름만으로 표현하면 길고 복잡해질 수 있다.

- 페이징과 함께 사용하기 좋음
Pageable과 함께 @Query를 사용하면 원하는 조건의 결과를 DB에서 바로 잘라서 가져올 수 있다.

- 의도가 명확함
Repository에서 어떤 조건으로 데이터를 가져오는지 한눈에 보이기 때문에 가독성과 유지보수 측면에서도 장점이 있다.

이번 미션에서는 단순히 기능이 동작하는 것뿐만 아니라, 도메인 관계를 어떻게 자연스럽게 모델링할 것인지,
그리고 조회 성능과 코드 유지보수성을 어떻게 함께 가져갈 것인지를 중요하게 생각했다.

그래서 엔티티를 역할별로 분리하고, 연관관계를 명확히 설정하고, 서비스 계층으로 비즈니스 로직을 분리하고, 조회는 @Query와 페이징을 활용하는 구조로 구현했다.
