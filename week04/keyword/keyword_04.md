## 아키텍처 구조란 ?
### 1. 정의

아키텍처 구조란 시스템을 구성하는 구성 요소(Components)와 그 요소들 사이의 상호작용, 그리고 이를 지배하는 원칙(Principles)의 집합입니다.

### 2. 아키텍처 구조의 목적

- **복잡성 관리**: 시스템이 커져도 구조가 잡혀 있으면 관리가 쉽습니다.
- **의사소통**: 개발자, 기획자, 고객이 같은 설계도를 보고 대화할 수 있습니다.
- **품질 결정**: 보안, 성능, 수정 용이성(유지보수) 등이 이 단계에서 결정됩니다.

### 3. 아키텍처 설계의 5대 원칙 (SOLID 성격)

- **모듈화 (Modularity)**: 시스템을 독립적인 단위(모듈)로 나눕니다. 하나가 고장 나도 전체가 멈추지 않게 하기 위함입니다.
- **추상화 (Abstraction)**: 복잡한 내부 구현은 숨기고 필요한 인터페이스만 드러냅니다.
- **계층화 (Layering)**: 비슷한 기능끼리 묶어 층을 쌓습니다. 상위 계층은 하위 계층만 사용하도록 제한하여 의존성을 줄입니다.
- **결합도 최소화 (Low Coupling)**: 모듈 간의 연결 고리를 느슨하게 만듭니다. 하나를 수정할 때 다른 모듈이 영향을 덜 받게 합니다.
- **응집도 최대화 (High Cohesion)**: 하나의 모듈 안에는 연관된 기능들만 모아둡니다**.**

### **4. 아키텍처 구조의 종류**

#### 계층형 아키텍처(Layered Architecture)

가장 전통적이고 많이 쓰이는 구조입니다.

- **구조**: 보통 `Presentation(화면) - Business(로직) - Data(DB)` 순으로 층을 나눕니다.
- **장점**: 구조가 단순하고 각 층별로 역할이 명확해서 관리가 쉽습니다.

#### 클라이언트-서버 구조 (Client-Server)

우리가 사용하는 웹 서비스의 기본 모델입니다.

- **구조**: 서비스를 요청하는 클라이언트와 서비스를 제공하는 서버로 나뉩니다.
- **장점**: 데이터 처리를 서버에서 집중적으로 할 수 있어 효율적입니다.

#### 마이크로서비스 아키텍처 (MSA)

요즘 넷플릭스나 쿠팡 같은 큰 서비스들이 쓰는 방식입니다.

- **구조**: 거대한 서비스 하나를 만드는 대신, 아주 작은 서비스(로그인, 결제, 배송 등) 여러 개를 만들어 연결합니다.
- **장점**: 특정 기능만 업데이트하거나 확장하기 매우 유리합니다.

#### MVC 패턴 (Model-View-Controller)

사용자 인터페이스(UI)가 있는 앱에서 필수적인 구조입니다.

- **Model**: 데이터와 비즈니스 로직.
- **View**: 사용자에게 보여지는 화면.
- **Controller**: 사용자의 입력을 받아 모델과 뷰를 연결하는 중개자.

## Swagger란 ?
### Swagger의 정의

- Swagger는 REST API를 설계, 빌드, 문서화하고 소비하는 데 도움을 주는 오픈소스 프레임워크입니다. 현재는 OpenAPI Specification(OAS)*이라는 표준 사양을 기반으로 동작하며, 개발자가 만든 코드를 분석해 웹 화면으로 예쁘게 보여주는 역할을 합니다.

### **Swagger 사용 목적**

1. **API 문서 자동화 (Documentation)**:
    - 과거에는 API를 만들고 나면 엑셀이나 Word에 따로 "이 주소로 데이터를 보내면 이런 답이 옵니다"라고 적어야 했습니다.
    - Swagger를 쓰면 코드에 주석 몇 줄만 달아도 **웹 페이지 형태의 문서**가 자동으로 생성됩니다.
2. **직접 테스트 가능 (Interactive Console)**:
    - 문서를 보는 것에서 끝나지 않고, 웹 화면에서 직접 "데이터 보내기(Try it out)" 버튼을 눌러 서버가 어떻게 응답하는지 실시간으로 확인할 수 있습니다. 포스트맨(Postman) 같은 도구 없이도 말이죠.
3. **협업 효율화**:
    - 백엔드 개발자가 API를 만들면, 프론트엔드 개발자는 Swagger 화면만 보고 "아, 이 데이터를 쓰면 되겠구나"라고 즉시 파악할 수 있어 소통 비용이 획기적으로 줄어듭니다.

### Swagger의 구성 요소

- **Swagger UI**: API 명세서를 브라우저에서 시각적으로 보여주는 도구 (가장 흔히 보는 화면).
- **Swagger Editor**: API 설계를 YAML이나 JSON 형식으로 편집하는 도구.
- **Swagger Codegen**: API 명세서를 기반으로 클라이언트 라이브러리나 서버 스터브 코드를 자동으로 생성해주는 도구.

## 도메인형 아키텍처란 ?
### 정의

- 도메인형 아키텍처는 소프트웨어의 복잡성을 해결하기 위해 비즈니스 로직(Domain)을 기술적인 코드(UI, DB 연결 등)로부터 완전히 분리하여, 시스템의 핵심을 보호하고 유연하게 만드는 구조입니다.

### 핵심 원칙

1. **비즈니스 로직 중심 (Focus on Domain)**:
    - 데이터베이스가 무엇인지, UI가 웹인지 앱인지는 나중 문제입니다. "주문이 들어오면 재고를 줄인다" 같은 비즈니스 규칙이 아키텍처의 심장부(Core)에 위치합니다.
2. **계층 간의 독립성 (Separation of Concerns)**:
    - 도메인 계층은 외부 환경(DB, API 등)에 의존하지 않습니다. 덕분에 DB를 Oracle에서 MySQL로 바꿔도 핵심 비즈니스 로직은 수정할 필요가 없습니다.
3. **유비쿼터스 언어 (Ubiquitous Language)**:
    - 개발자와 기획자, 현업 담당자가 똑같은 용어를 사용해 설계합니다. 코드를 보면 실제 업무 프로세스가 그대로 읽히도록 만드는 것이 목표입니다.

### 대표적인 종류 (아키텍처 스타일)

도메인 중심 설계를 실현하기 위한 구체적인 모양들입니다.

1. **헥사고날 아키텍처 (Hexagonal Architecture)**:
    - '포트와 어댑터' 구조라고도 합니다. 중심에 도메인을 두고, 외부 시스템(DB, UI)은 어댑터를 통해 탈부착하듯이 연결합니다.
2. **클린 아키텍처 (Clean Architecture)**:
    - 원형 구조로, 안쪽(도메인)으로 갈수록 고수준의 규칙을 담고 의존성은 항상 안쪽을 향하게 설계합니다.
  
## DDD vs 도메인형 아키텍처
### DDD (Domain-Driven Design, 도메인 주도 설계)

DDD는 소프트웨어를 만들 때 기술보다 '비즈니스 도메인'의 복잡성을 해결하는 데 집중하자는 방법론 또는 철학입니다.

- **전략적 설계 (Strategic Design)**: 비즈니스를 큰 덩어리(Bounded Context)로 나누고, 각 팀이나 모듈이 맡을 경계를 정하는 것 (지도 그리기).
- **전술적 설계 (Tactical Design)**: 그 안에서 실제로 코드를 어떻게 짤지 결정하는 것 (Entity, Aggregate, Service 등 구체적인 도구 사용).
- **핵심**: 개발자만 아는 용어가 아니라, 기획자/현업과 공통된 언어(**Ubiquitous Language**)를 사용하여 코드를 짜야한다. 라는 것.

### 도메인형 아키텍처 (Domain-Driven Architecture)

DDD라는 철학을 실제 소프트웨어 구조로 옮겨놓은 기술적인 틀입니다.

- 비즈니스 로직(Domain)이 가장 안쪽에 있고, 데이터베이스나 UI 같은 기술적 요소는 바깥쪽(Infrastructure)에 위치하여 서로 영향을 주지 않게 격리합니다.

| **구분** | **DDD (방법론)** | **도메인형 아키텍처 (구조)** |
| --- | --- | --- |
| **성격** | 사고방식, 설계 원칙, 협업 방식 | 실제 코드의 배치 구조, 계층 분리 |
| **목표** | 비즈니스 복잡성을 이해하고 정복함 | 비즈니스 로직을 기술로부터 보호함 |
| **도구** | 유비쿼터스 언어, 컨텍스트 맵 | 레이어, 포트와 어댑터, 의존성 역전 |

**한 줄 정리
- DDD는 비즈니스 도메인을 중심으로 설계를 주도하는 방법론이며, 도메인형 아키텍처는 이러한 DDD의 원칙을 물리적인 코드 구조로 구현한 아키텍처 양식이라고 할 수 있습니다.**

## 왜 DTO를 사용하는가 ?
### 1. DTO의 정의

DTO(Data Transfer Object, 데이터 전송 객체)는 계층 간 데이터 교환을 위해 만들어진 객체입니다. 로직(비즈니스 규칙)을 전혀 가지지 않고, 오직 데이터만 담고 있는 바구니 같은 역할을 합니다.

### 2. DTO를 사용하는 이유

#### 엔티티(Entity) 보호 및 캡슐화

- **상황**: 데이터베이스와 직접 연결된 '엔티티' 객체에는 비밀번호나 주민번호 같은 민감한 정보가 포함될 수 있습니다.
- **해결**: 외부(UI/API)에 데이터를 보낼 때 엔티티를 그대로 노출하지 않고, 필요한 정보만 골라 담은 DTO를 보내 보안을 강화합니다.

#### 계층 간 결합도 낮추기 (Decoupling)

- **상황**: 만약 화면(View)에서 요구하는 데이터 형식이 바뀐다고 해서 DB 구조(Entity)까지 바꿔야 한다면 매우 비효율적입니다.
- **해결**: DTO가 중간에서 완충 작용을 해줍니다. DB 구조가 바뀌어도 DTO만 적절히 변환해주면 화면 쪽 코드는 수정할 필요가 없습니다.

#### 네트워크 전송 최적화

- **상황**: 화면에서 "이름"만 필요한데, 서버에서 "주소, 전화번호, 가입일"까지 다 담긴 무거운 객체를 보내면 네트워크 낭비입니다.
- **해결**: 딱 필요한 데이터만 추려서 전송하므로 호출 횟수와 데이터 양을 줄일 수 있습니다.

#### 데이터 유효성 검증(Validation)의 분리

- **상황**: 화면에서 들어오는 데이터는 "필수 입력값" 검증이 필요하고, DB에 저장될 때는 "중복 여부" 검증이 필요합니다.
- **해결**: DTO에 `@NotNull` 같은 검증 로직을 넣어, 비즈니스 로직이 담긴 엔티티와 검증 로직을 분리할 수 있습니다.

- DTO는 내부의 핵심 모델인 엔티티를 외부로부터 보호하고, 각 계층이 서로의 세부 구현에 얽매이지 않도록 데이터만 순수하게 전달하기 위해 사용합니다.

## 컨버터는 왜 사용하는가 ?
### 1. 컨버터의 정의

컨버터는 두 개의 서로 다른 데이터 모델(예: Entity와 DTO, 혹은 String과 Integer) 사이에서 데이터 형식이나 구조를 일치시키기 위해 변환 로직을 수행하는 객체나 함수를 말합니다.

### 2. 컨버터를 사용하는 이유

#### 비즈니스 로직과 변환 로직의 분리

- **상황**: 서비스 코드(`Service Layer`) 안에 "엔티티를 DTO로 바꾸는 코드"가 여기저기 섞여 있으면 가독성이 떨어집니다.
- **해결**: 변환만 담당하는 '컨버터'를 따로 두어, 서비스 코드는 순수하게 비즈니스 로직(예: 계산, 저장 등)에만 집중하게 합니다.

#### 코드 중복 방지 (Reusability)

- **상황**: '사용자 정보'를 보여주는 화면이 여러 개일 때, 매번 엔티티를 DTO로 바꾸는 코드를 작성하는 건 비효율적입니다.
- **해결**: 잘 만들어진 컨버터 하나를 여러 곳에서 재사용하여 코드 양을 줄이고 실수를 방지합니다.

#### 데이터 타입 불일치 해결

- **상황**: 데이터베이스에는 날짜가 `Timestamp` 형식으로 저장되어 있는데, 화면에는 `2026-04-03` 같은 `String` 형식이 필요할 때가 있습니다.
- **해결**: 컨버터가 중간에서 형식을 가공하여 양쪽 시스템이 원하는 데이터를 정확히 받을 수 있게 해줍니다.

DTO를 데이터를 담는 ‘상자’라고 생각한다면, 컨버터는 상자 안의 물건을 다른 상자로 옮겨 담는 ‘사람(기능)’이라고 쉽게 이해할 수 있습니다.
